# PLAN2 - Adaptive LLM-Generated Site Parsers

## Problem Statement

Every job board has its own DOM structure. Maintaining hand-written scrapers per site
(LinkedIn bpr-guid JSON, Seek data-automation attributes, etc.) is fragile - sites
change their markup without notice, and each new job board requires custom code.

## Proposed Solution

Replace per-site scraper code with **LLM-generated parser scripts** that are cached
on disk. The companion uses a cached parser when available (fast), falls back to LLM
generation when the parser is missing or broken (slow but self-healing).

---

## Architecture Overview

```
Extension (popup.js)
  scrapeJobFromPage() - simplified to just grab document.body.innerText
  POST /extract  { domain, page_text }  →  companion

Companion (/extract endpoint)
  1. Normalize domain  (e.g. "www.linkedin.com" → "linkedin.com")
  2. Look for  ~/.overhired/parsers/linkedin.com.py
  3a. Found:   run parser(page_text) → { title, company, description, location }
               if parser raises: fall back to step 4
  3b. Missing: go to step 4
  4. LLM extraction:
       - Send page_text to local LLM with extraction prompt
       - LLM returns structured JSON  AND  a Python parser function
  5. Save parser to ~/.overhired/parsers/{domain}.py
  6. Run saved parser to verify output
  7. Return { title, company, description, location }
```

## Parser File Format

Each parser is a single Python file with a standard interface:

```python
# ~/.overhired/parsers/linkedin.com.py
# Auto-generated by overhired companion on 2026-05-12
# Domain: linkedin.com
# Regenerate: delete this file and scan again

def extract(text: str) -> dict:
    """Extract job info from linkedin.com page innerText."""
    lines = [l.strip() for l in text.splitlines() if l.strip()]

    title   = ''
    company = ''
    desc    = ''

    # LLM-discovered patterns for this domain
    for i, line in enumerate(lines):
        if 'About the job' in line:
            desc = '\n'.join(lines[i+1:i+200])
            break

    if len(lines) > 0: title   = lines[0]
    if len(lines) > 2: company = lines[2]

    return {
        'title':       title,
        'company':     company,
        'description': desc[:6000],
        'location':    '',
    }
```

The interface contract is fixed:
- Input:  `text: str`  (document.body.innerText, capped ~12KB)
- Output: `dict` with keys `title`, `company`, `description`, `location`
- Any exception = parser is broken, trigger regeneration

### New files in companion/

```

  parser_runner.py  - sandboxed exec() of parser .py files
  parsers/          - auto-generated parser scripts (gitignored)

    seek.co.nz.py
```

### extractor.py logic (pseudocode)

```python
PARSERS_DIR = Path("~/.overhired/parsers").expanduser()

def extract(domain: str, page_text: str) -> dict:
    parser_path = PARSERS_DIR / f"{domain}.py"

    # 1. Try cached parser
    if parser_path.exists():
        result = run_parser(parser_path, page_text)
        if result and result.get('title'):
            return result
        # Parser failed or returned empty - regenerate
        parser_path.unlink()

    # 2. LLM extraction + parser generation
    result, parser_code = llm_extract_and_generate(domain, page_text)

    # 3. Save parser for next time
    PARSERS_DIR.mkdir(parents=True, exist_ok=True)
    parser_path.write_text(parser_code)

    return result
```

### parser_runner.py (sandboxed exec)

```python
def run_parser(parser_path: Path, page_text: str) -> dict | None:
    code = parser_path.read_text()
    namespace = {}
    try:
        exec(compile(code, str(parser_path), 'exec'), namespace)
        extract_fn = namespace.get('extract')
        if not callable(extract_fn):
            return None
        result = extract_fn(page_text)
        # Validate output shape
        if isinstance(result, dict) and result.get('title'):
            return result
    except Exception as e:
        print(f"[overhired] Parser {parser_path.name} failed: {e}")
    return None
```

## LLM Prompt for Parser Generation

The prompt asks the LLM to do two things in one call:
1. Extract the job info from this specific page text (immediate result)
2. Write a reusable Python parser function for this domain (cached for later)

```
System:
  You are a web scraping expert. Given the visible text of a job listing page,
  extract structured job information AND write a reusable Python parser function.

User:
  Domain: {domain}

  Page text (document.body.innerText):
  {page_text}

  Tasks:
  1. Extract: title, company, description, location from this page text.
  2. Write a Python function  extract(text: str) -> dict  that would reliably
     extract the same fields from ANY job page on {domain}.
     - Use text patterns you observed (line positions, surrounding keywords, etc.)
     - Do not hardcode specific values from this page
     - Return dict with keys: title, company, description, location

  Respond in this exact JSON format:
  {
    "extracted": {
      "title": "...",
      "company": "...",
      "description": "...",
      "location": "..."
    },
    "parser_code": "def extract(text: str) -> dict:\n    ..."
  }
```

## Extension Changes

### scrapeJobFromPage() - drastically simplified

The current 200-line function with per-site heuristics becomes ~15 lines:

```javascript
function scrapeJobFromPage() {
  // Just grab visible text - companion + LLM handle the parsing
  const text = document.body.innerText.slice(0, 12000);
  const domain = window.location.hostname.replace(/^www\./, '');
  const ats = detectATS(); // keep ATS detection here (URL-based, instant)
  return { domain, page_text: text, ats };
}
```

### scanPage() change

Instead of using the scraped result directly, call `/extract` first:

```javascript
const scanPage = useCallback(async () => {
  setScanState('scanning');
  const [tab] = await chrome.tabs.query({ active: true, currentWindow: true });
  const results = await chrome.scripting.executeScript({
    target: { tabId: tab.id },
    func: scrapeJobFromPage,
  });
  const raw = results[0]?.result;

  // Send to companion for LLM-powered extraction
  const resp = await fetch(`${companionUrl}/extract`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ domain: raw.domain, page_text: raw.page_text }),
  });
  const j = await resp.json();
  setTitle(j.title || '');
  setCompany(j.company || '');
  setDesc(j.description || '');
  setScanState('found');
}, []);
```

## Reliability & Fallback Chain

```
1. Cached parser exists + returns non-empty title    → use it  (fast, ~0ms)
2. Cached parser missing or broken                   → LLM extraction (slow, ~5-30s)
3. LLM extraction succeeds                           → save parser, use result
4. LLM extraction fails / malformed JSON             → return raw innerText best-effort
5. Everything fails                                  → show error, user types manually
```

The editable Role/Company fields in the popup remain as the final escape hatch.

---

## Parser Lifecycle

| Event | Action |
|-------|--------|
| First scan of new domain | LLM generates + saves parser |
| Subsequent scans (parser works) | Run parser directly, skip LLM |
| Site changes structure (parser returns empty) | Delete parser, re-run LLM |
| User manually triggers regeneration | `DELETE /parsers/{domain}` endpoint |
| Parser file corrupted | Caught by exception handler, treated as missing |

Parsers live in `~/.overhired/parsers/` - gitignored, user-local.
The companion startup log should report which parsers are cached.

## Optional Phase 2 - Agentic Loop via MCP

For higher reliability (~95% vs ~70% one-shot), add an MCP server that gives the
LLM tool access to test and fix its own generated parser before saving.

MCP tools the LLM would have:
- `run_code(code, input)` - execute the parser function with sample input, see output
- `read_file(path)` - read an existing parser
- `write_file(path, content)` - save a parser
- `list_parsers()` - see what's cached

The agentic loop:
```
LLM generates parser
  → calls run_code(parser, page_text)
  → sees output
  → if wrong: fixes and retries (up to 3 iterations)
  → if right: calls write_file to save
```

This requires the companion to implement MCP server protocol (JSON-RPC over stdio or
HTTP) or use an existing MCP SDK. Complexity is significantly higher but the quality
of generated parsers improves substantially.

Skip this in Phase 1. Implement only if one-shot generation is unreliable in practice.

## Notes

- `document.body.innerText` is universally available on all job boards
- Cap page_text at 12KB before sending - enough for any job description, avoids
  overwhelming the LLM context window
- Parsers are plain Python with no dependencies - safe to exec() in a namespace
- The /extract endpoint should be fast to respond on cache hit (no LLM involved)
- Generated parsers should be human-readable and easy to hand-edit when needed
- Consider adding a "Regenerate parser" button in the extension Settings tab
